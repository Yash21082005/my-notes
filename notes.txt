Here's a structured list you can use.

1. Understand the overall architecture

Start with:

"Can you explain the current PostgreSQL HA architecture and the role of NGINX in it?"

Then ask:

How many PostgreSQL nodes do we have?
Is it one Primary and multiple Replicas?
Which replication method are we using (streaming replication, logical replication, etc.)?
Are we using Patroni or another HA solution?
2. Ask specifically about NGINX

This is the most important part.

Ask:

"How does NGINX know whether a query is a read or a write?"

Then follow up with:

Does NGINX inspect SQL queries?
Or are there separate endpoints for reads and writes?
Does the application send read traffic to one endpoint and write traffic to another?
Does NGINX only load balance connections, or does it perform query routing?
3. Ask about failover
If the Primary database goes down, what happens?
Who elects the new Primary?
Does NGINX automatically detect the new Primary?
How long does failover usually take?
Does the application need any configuration changes after failover?
4. Ask about Replicas
How many replicas do we have?
Are all read requests distributed among replicas?
What load-balancing algorithm does NGINX use (Round Robin, Least Connections, etc.)?
If one replica fails, what happens?
5. Ask about consistency
What happens if a user writes data and immediately performs a read?
Can the replica return stale data because of replication lag?
How do we handle that scenario?
6. Ask about the application
Does the Spring Boot application know about the Primary and Replicas?
Does it use one database URL or multiple URLs?
Where is the routing configured—in the application or in the infrastructure?
7. Ask about monitoring
How do you monitor replication lag?
How do you know if a replica is unhealthy?
Which tools or dashboards do you use?
8. Ask about NGINX configuration
Can you show me the NGINX configuration used for PostgreSQL?
How are the upstream servers configured?
How does health checking work?
Questions that will impress them
"How do we prevent reads from going to the Primary unnecessarily?"
"How do we ensure no writes ever reach a Replica?"
"What happens to existing client connections during failover?"
"How is replication lag monitored, and what's considered acceptable?"
"What are the biggest challenges you've faced with this setup?"
One thing to avoid

Don't say:

"ChatGPT told me NGINX cannot do this."

Instead, ask:

"Can you help me understand how our setup handles read/write routing? I'd like to understand where that decision is made."

That keeps the discussion collaborative and lets them explain the specifics of your company's architecture.
